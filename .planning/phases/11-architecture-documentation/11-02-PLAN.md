Replace the current §7 body ("## 7. Assembly — split: build-time vendor/sync (committed) + baked images") with a description of the Phase 8 Dockerfile recipe model. The new §7 must cover:

1. **Recipe = Dockerfile body.** A recipe contributes a `Dockerfile` (no `FROM` line) that the assembler concatenates into the derived stack Dockerfile. `ARG HARNESS` at the top of each recipe Dockerfile lets the body reference `${HARNESS}` for harness-specific installs (e.g. `pnpm dlx @gstack/install --host ${HARNESS}`). The assembler strips `ARG HARNESS` lines and emits `FROM harnessed-${HARNESS}:latest` as the header before concatenating bodies.

2. **recipe.yaml declares metadata, not build steps.** The `recipe.yaml` still carries `name`, `description`, `mcp.servers` (for hatago config), and `harnesses:` (compat list) + `expect:` (smoke-check entries). The build steps live in the `Dockerfile`, not in YAML fields like `deps`, `plugins`, or `hooks`. The old `vendor-plugin`/`sync-plugin-links` approach is superseded.

3. **Supply chain = pin sources + scan the derived image.** Recipes pin their sources inside the Dockerfile body (pinned `git clone --branch <tag>`, pinned `pnpm dlx @pkg@<exact-version>`, or `ARG`-driven version pins). The assembler validates that no floating ref (`--branch main`, unversioned `pnpm dlx @pkg`) passes. After building the derived image `harnessed-<stack>`, osv-scanner V2 scans the image (not vendored source files). The "scan the derived image" approach catches transitive CVEs that a source-only scan misses.

4. **Assembler output.** The assembler emits `profiles/<stack>/Dockerfile.harnessed-<stack>` (the concatenated derived Dockerfile) and `profiles/<stack>/hatago.config.json` (from recipe.yaml `mcp.servers` entries). The host then runs `podman build` on the emitted Dockerfile to produce `harnessed-<stack>`.

5. **pnpm everywhere** — the supply-chain policy sub-section stays largely the same; update any references to "vendored plugins" → "recipe Dockerfiles", "vendor-plugin" → Dockerfile pinning.

Keep the heading "## 7. Assembly" but retitle the sub-heading to match the new model. Retain the Scanner credentials sub-section (snyk/Socket.dev policy is unchanged). Remove the old "Prior-art source" table with vendor-plugin/sync-plugin-links, as those are no longer the mechanism.

Replace the current §18 body (starting "## 18. Proposed: testing — integration only, behavior through the instance") with a description of the Phase 10 two-oracle capability test. The new §18 must cover:

1. **Two-oracle approach.** The capability test uses two complementary checks that together give confidence the stack exposes exactly what its manifest declares:
   - **Oracle 1 — Structured MCP probe (deterministic).** Query hatago's `hatago://servers` resource (a JSON snapshot of connected servers) directly — no model call needed. Assert the manifest's `mcp.servers` entries appear in the response. This is fast and deterministic.
   - **Oracle 2 — Un-primed agent probe (behavioral).** Ask the harness, headless, what capabilities it has — deliberately NOT priming it with the expected list. The probe asks the agent to enumerate available skills and MCP servers. Assert declared capabilities appear and a known-absent "decoy" capability does NOT appear.

2. **Negative control (anti-sycophancy gate).** The decoy capability is a name that exists in neither the recipe nor the image. If the agent claims the decoy is present, the test exits non-zero with status INVALID (distinct from a capability-failure non-zero exit). The INVALID banner appears in `capability-report.md`. This catches sycophantic priming — a model hallucinating agreement with a leading question.

3. **Capability report.** After every `harnessed test <stack>` run, `profiles/<stack>/capability-report.md` is written with a per-capability table showing ✓/✗ for each MCP server and `expect:` entry, plus the INVALID banner when priming is detected.

4. **Philosophy unchanged.** Retain the existing rationale: test behavior through the public interface (the running instance), the stack manifest is the test oracle, no assembler unit tests. Update any language that says "proposed" to "implemented."

5. **Remove stale details.** The v1 description asked the harness to "emit the skills/commands it sees as JSON and diff against the manifest" — this is superseded. Remove the old step-by-step list that mentions "ask the harness…to emit skills/commands…as JSON."

Keep "## 18." heading; retitle from "Proposed: testing" to just "Testing — integration only, behavior through the instance."

Update §4b of docs/harnessed-design.md (the "Config source" subsection under "## 4. Mounts") to describe the Phase 9 surgical per-file profile mount model. The existing §4b text states that `skills/`, `commands/`, `agents/`, `hooks/`, `rules/`, `.mcp.json`, `settings.json` "come only from the stack profile (§7)" — update this to explain HOW they are mounted: individually, via per-harness YAML manifest, not as a whole directory.

The updated §4b must cover:

1. **Surgical per-file mounting.** Profile assets are NOT mounted as a whole `profiles/<stack>/` directory. Instead, Phase 9 introduced per-harness YAML manifests (`lib/manifests/<harness>.yaml`) that enumerate exactly which files to mount. Each manifest has two top-level keys:
   - `profile_files`: individual filenames (e.g. `.mcp.json`, `settings.json`) to mount read-only from `profiles/<stack>/` into the container's harness config directory (e.g. `~/.claude/<f>` for claude/omp/opencode).
   - `history_dirs`: `$HOME`-relative paths to bind read-write for history surfacing (e.g. `.claude/projects`, `.claude/todos` for claude).

2. **Harness-aware container targets.** The bash helper `lib/harnessed-manifest-mounts.sh` reads the manifest at launch time and applies harness-aware container target paths: claude/omp/opencode mount profile files to `~/.claude/<f>`; gemini/antigravity/codex skip profile file mounting (their config is baked in the image). This means the profile directory structure is flat — per-file, not a subtree.

3. **Skills and SKILL.md entries.** Skills (`.claude/skills/<name>/SKILL.md`) are also per-file mounts derived from the profile; none come from the host's own `~/.claude/` tree.

4. **Credentials are separate (§4a, not manifests).** The `~/.claude/.credentials.json` auth mount is in the §4a host-integration layer — it is NOT in the per-harness manifest. Manifests list only config/skill files, never auth credential paths.

5. **Why surgical?** Mounting a whole directory would let host defaults (other skills, stale config, personal CLAUDE.md) bleed into the container — exactly the "no host defaults" invariant §4 is designed to protect. Individual file mounts make the container's config surface exactly what the profile declares, no more.

Update the heading of §4b to: `### 4b. Config source — surgical per-file mounts`

Retain all existing sentences about `.credentials.json` (OAuth, mount ro), the `.claude.json` stub warning (do not mount, generate minimal stub), and session state persistence (`~/.claude/projects/`). Insert the surgical mount explanation as a new paragraph/bullets after the stub sentence and before the session state bullet.

Do NOT change §4a or any other section of harnessed-design.md in this task.
