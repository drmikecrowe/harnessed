# Phase 4 Plan 02: State Persistence + Full CLI Summary

**Isolated stacks now persist session state by default (wiped only under `--fresh`), host-side under XDG with a legible slug — plus the full operable CLI: `list`/`stop`/`rm` subcommands, `new` scaffolding, and `install`/`uninstall` launcher shims.**

## Performance
- **Duration:** ~45 min (executor auto-tasks + checkpoint verification)
- **Tasks:** 4 auto + 1 human-verify checkpoint (approved via podman: persistence marker + CLI round-trips)
- **Files:** 6 (1 created + 5 modified... effectively 3 touched: harnessed-cli.sh NEW, harnessed-isolated.sh, harnessed)

## Accomplishments
- State persistence (STA-01/02): the copy-on-start wipe is gated behind `--fresh`/first-create — a normal recreate reuses accumulated `~/.claude` (projects/, history.jsonl); `--fresh` wipes. Proven: a marker survives a non-`--fresh` recreate and is gone after `--fresh`. Session state is host-side under `$XDG_STATE_HOME/harnessed/<instance>/` (legible slug, harnessed-scoped).
- CLI-01: `list` (stacks + instances), `stop <stack>` / `rm <stack>` (by stack name, all instances across projects); legacy `--list/--stop/--remove` retained.
- CLI-02: `new <stack> [--harness claude|omp] [--recipes a,b,c]` scaffolds `stacks/<stack>/stack.yaml`, validates the harness, refuses overwrite.
- CLI-03: `install <stack>` / `uninstall <stack>` write/remove a `~/.local/bin/<stack>` shim (the §13 template: `exec $HARNESSED_PATH <stack> "$@"`, +x).
