<objective>
Add the second harness — `omp` (Oh My Pi) — consuming the same Claude-canonical profile via the
`claude-hooks-bridge` extension, and prove recipe breadth with a second recipe (HRN-01).

Purpose: today `HARNESS_CONFIG_DIR = {claude: .claude}` — an `omp` stack is a hard schema error, there is no
`harnessed-omp` image, and the launcher/capability-test only know `claude`. The design (§6/§8) specifies omp
as a peer harness that consumes Claude-format skills/hooks/commands at runtime via the bridge — no
re-authoring. Recipe breadth (success criterion 2: "a second recipe added to a stack is exposed and verified")
is proven alongside: a multi-recipe claude stack asserts two recipes compose.

Output: `base/Dockerfile.harnessed-omp` (the omp image + pre-installed bridge) + schema omp mapping + a
`greet` skill recipe + an `omp` base recipe + `claude-multi` and `omp-time` stacks + harness-aware launcher
dispatch + a harness-aware capability test. The omp-specific unknowns (skill discovery, MCP wiring, headless
auth — design §14 "to verify") are resolved in the checkpoint.
</objective>

<threat_model>
**ASVS L1 — block on high. A new harness image + a bridge that executes hooks are the new surfaces.**
- **T-04-09 (Tampering/Supply-chain, omp + bridge images):** the omp image installs omp (mise) + the bridge (npm). Mitigation: the Phase 3 scan gate (BLD-02) applies to the `harnessed-omp` image on build (reuse build_stack's image-scan path); the bridge is a pinned published npm package (`@drmikecrowe/omp-claude-hooks-bridge`, MIT, owner-authored). Disposition: mitigate.
- **T-04-10 (Elevation/Injection, bridge executes hooks via `bash -lc`):** the claude-hooks-bridge runs `.claude/settings.json` hook commands via `bash -lc` — a malicious hook executes arbitrary code inside the instance. Mitigation: this is the SAME threat surface as Claude's own hooks (the profile is authored/committed, reviewed at assemble time); the isolated instance is sandboxed (egress firewall, no host config layer); hooks are not a NEW privilege escalation beyond what Claude already offers. Disposition: accept (parity with claude harness; isolation contains it).
- **T-04-11 (Spoofing, omp headless auth):** omp `--profile` isolates auth; the credentials mount (Phase 2 AUTH-02) seeds auth. If omp prompts for auth headless, the test fails loud. Mitigation: verify omp boots headless with the seeded credentials (P-04-12); no token is baked. Disposition: mitigate (verify in checkpoint).
- **T-04-SC (Supply-chain, new packages):** NEW = omp (mise/github:can1357/oh-my-pi, the host harness) + `@drmikecrowe/omp-claude-hooks-bridge` (owner-authored npm, MIT). Both are first-party/owner-controlled, MIT, scanned on image build. No `[SUS]`/`[SLOP]` package. Disposition: mitigate (scan on build).
</threat_model>
