<objective>
Add the cross-instance concurrency model: shared service sidecars that are their own image/container/volume
with a lifecycle independent of any instance, managed by `harnessed svc up|down|list`, and concurrently
attachable by multiple isolated instances over `harnessed-net` (SVC-01, SVC-02, SVC-03).

Purpose: today every isolated stack is a self-contained pod (harness + hatago) with no shared, stateful
sidecar. Heavy services (hindsight, openbrain) need to be a SINGLE long-lived container that many instances
attach to concurrently — service-scoped (one memory across `claude+hindsight` and `omp+hindsight`), not
stack-scoped. The design (§3/§9) specifies this as standalone containers on a shared network, but
`harnessed-net` is currently opt-in (default pasta) and there is no `services/` model, no `svc` command,
and no assembler resolution of `service:`-referenced MCP servers.

Output: `lib/harnessed-services.sh` (the service lifecycle) + `services/ping/` (a lightweight network-native
MCP tracer service — FastMCP streamable-http on :8080) + `recipes/ping/` + `stacks/ping-time/` + schema
`load_service` + assembler service→url resolution + `harnessed-net` as default + the `svc up|down|list`
command group + a concurrency proof (two instances, one `ping` container).
</objective>

<threat_model>
**ASVS L1 — block on high. A shared service on harnessed-net is a new network-reachable surface.**
- **T-04-01 (Information disclosure / cross-tenant, shared-netns eavesdrop):** two instances on `harnessed-net` could reach each other's containers or the service's volume data. Mitigation: service volumes are service-scoped named volumes mounted ONLY into the service container (never into instances); instances reach the service over HTTP at its MCP port only (the service does not expose its data path); no instance mounts another instance's state dir. Disposition: mitigate.
- **T-04-02 (Tampering/Supply-chain, malicious service image):** a service image built from `services/<name>/Dockerfile` is never scanned. Mitigation: the Phase 3 scan gate (BLD-02) applies to service images — `svc up` runs the host `podman save` + `scan-image` gate on first build (reuse `build_stack`'s image-scan path). Disposition: mitigate.
- **T-04-03 (Denial of service, service crash takes down instances):** an instance depends on a service that crashes; hatago reports it disconnected and the capability test fails. Mitigation: this is the correct behavior (fail loud, not silent) — the service has a healthcheck and `svc up` waits for it; an instance does NOT own the service lifecycle (design §9). Disposition: accept (correct semantics).
- **T-04-04 (Elevation, service container runs as root / privileged):** Mitigation: service containers run `--userns=keep-id` (unprivileged), no `--privileged`, no daemon socket. Disposition: mitigate.
- **T-04-SC (Supply-chain, new packages):** NEW this plan = `mcp` (Anthropic official Python SDK, PyPI) for the ping server. RESEARCH legitimacy: **[OK]** — `mcp` is Anthropic's official MCP SDK, widely used, MIT licensed. The `python:3.12-slim` base is the canonical slim Python image. The ping service image is scanned by BLD-02 on build. No `[SUS]`/`[SLOP]` package → no blocking checkpoint. Disposition: mitigate (scan on build).
</threat_model>
