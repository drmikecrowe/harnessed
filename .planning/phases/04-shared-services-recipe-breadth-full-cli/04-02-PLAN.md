<objective>
Make isolated stacks persistent by default (with `--fresh` for clean-room runs), land host-side session
persistence, and deliver the full operable CLI surface (`list`/`stop`/`rm`, `new`, `install`/`uninstall`)
that the design §13 specifies (STA-01, STA-02, CLI-01, CLI-02, CLI-03).

Purpose: today `harnessed_isolated` does `rm -rf` + copy-on-start on EVERY create (line 81), so an instance
never accumulates state — a memory system's value is its accumulation, and `--fresh` is currently identical
to a normal run (both wipe). The CLI is also flag-only (`--list/--stop/--remove`); the §13 surface wants
first-class subcommands plus stack scaffolding and the install shim that lets you launch a stack by name
from anywhere.

Output: a ~5-line state-persistence fix in `harnessed_isolated` + `lib/harnessed-cli.sh` (new/install/uninstall/
list/stop/rm) + the launcher subcommand wiring. Small, high-leverage: persistence unblocks the memory-system
value prop, and the CLI makes stacks operable.
</objective>

<threat_model>
**ASVS L1 — block on high. State/CLI changes are low-risk but each has a failure mode.**
- **T-04-05 (Information disclosure, session history at rest):** persistent state now ACCUMULATES `projects/` + `history.jsonl` (potentially sensitive prompts/output) host-side. Mitigation: it lives under `$XDG_STATE_HOME/harnessed/<instance>/` (harnessed-scoped, user-owned, 0700 by default), NOT the host's own `~/.claude`; no new network exposure (host-side only). Disposition: accept (same posture as any local tool's state; documented location).
- **T-04-06 (Tampering/PATH injection via install shim):** `harnessed install` writes to `~/.local/bin/<stack>`. A malicious shim there would run on `<stack>` invocation. Mitigation: the shim is generated from a fixed template (no user-controlled content beyond the stack name + absolute path), written with the user's umask (not world-writable), and `uninstall` removes exactly that file. Disposition: mitigate.
- **T-04-07 (Denial of service / data loss, persistent-state wipe bug):** if the `--fresh` gate is inverted, a normal recreate nukes accumulated history (P-04-06). Mitigation: the acceptance test DISTINGUISHES the two paths (marker survives non-fresh, gone after fresh) — an inverted gate fails the test loudly. Disposition: mitigate.
- **T-04-08 (Tampering, `new` overwrites an existing stack):** Mitigation: `new` refuses to overwrite (file-exists check before write). Disposition: mitigate.
- **T-04-SC (Supply-chain):** no new packages this plan (pure bash + the existing schema). Disposition: accept (none introduced).
</threat_model>
