<objective>
Ship one managed pnpm supply-chain config and apply it everywhere pnpm runs, so every JavaScript install —
global, per-recipe, and hatago's bundled servers — is policy-governed (BLD-01).

Purpose: This plan is the first half of making `harnessed build` trustworthy. Today every image installs
JS via `pnpm@latest` with **no managed policy**, and mise's `npm:` backend silently uses npm for the global
node tools — so none of the three BLD-01 controls (release-age quarantine, lifecycle default-deny, store
integrity) are actually in effect. Pinning pnpm@11 and shipping one explicit, auditable config from `lib/`
flips all three v11 defaults on for real. Plan 03-02 then builds the scan gate and raw-npm lint on top of a
policy-governed install base.

Output: `lib/pnpm/config.yaml` (new) + pnpm@11 pin + mise routing + COPY in `base/Dockerfile.harnessed-base`,
`base/Dockerfile.hatago`, `tools/Dockerfile`, and the legacy `Dockerfile`; the stale `onlyBuiltDependencies`
wording in `docs/harnessed-design.md` §7 corrected to `allowBuilds`.
</objective>

<threat_model>
**ASVS L1 — block on high. This plan IS a security control (ASVS V10 Supply Chain).**
- **T-03-01 (Tampering/Supply-chain, image installs):** a compromised freshly-published npm release or a malicious lifecycle/postinstall script is installed into harnessed-base/hatago/tools/legacy because the policy is inert. Mitigation: ship `minimumReleaseAge: 1440` (release-age quarantine) + `minimumReleaseAgeStrict: true` (fail-closed) + `strictDepBuilds: true` + `allowBuilds` allowlist (lifecycle default-deny) + `verifyStoreIntegrity: true` + `blockExoticSubdeps: true`, pinned on pnpm@11. Disposition: mitigate (this plan implements it).
- **T-03-02 (Tampering, mise npm backend bypass):** without `npm.package_manager=pnpm`, mise's `npm:`-backend tools install via npm — no quarantine, no lifecycle deny — silently defeating BLD-01 for exactly the global tools. Mitigation: set `mise settings set npm.package_manager pnpm` BEFORE `mise install` in harnessed-base and the legacy Dockerfile. Disposition: mitigate.
- **T-03-03 (Configuration/Spoofing, inert policy):** authoring the config with the REMOVED v10 key `onlyBuiltDependencies` (ignored in v11) or putting non-auth policy in `.npmrc` (auth-only in v11) makes the guard appear configured but do nothing. Mitigation: use `allowBuilds` (map) ONLY; put policy in `pnpm-workspace.yaml`/global `~/.config/pnpm/config.yaml`, never `.npmrc`; verify with `pnpm config list` in the built image. Disposition: mitigate.
- **T-03-SC (Supply-chain, new packages):** NO new npm/pip/cargo package is introduced by this plan — pnpm@11 ships via mise (already in repo, approved), `@himorishige/hatago-mcp-hub` is pre-existing and pinned (Phase 2). Disposition: accept (config-only; no Package Legitimacy Gate checkpoint needed).
</threat_model>
