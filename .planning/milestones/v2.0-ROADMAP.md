## Overview

Replace the typed-YAML recipe model with a Dockerfile-based model where recipes run frameworks' own
installers. Rebuild the image lineage into 3 layers (harnessed-base → harnessed-<harness> → harnessed-<stack>).
Deliver surgical profile mounts so image-baked skills survive. Gate every derived build on pin
validation and osv-scanner V2. Update all architecture documentation to match the new model.

## Milestone Summary

**Key Accomplishments:**

1. Fat base image: harnessed-base now has all runtimes (node@24, bun, rust, go, python, pnpm@11) — no harness CLIs in base
2. Agent images: standalone cached `harnessed-claude` and `harnessed-omp` via `FROM harnessed-base + one CLI`
3. Dockerfile recipe model: recipes are Dockerfiles that run frameworks' own installers, parameterized by `--host ${HARNESS}`, pinned to tag/SHA
4. 3-layer derived images: assembler concatenates recipe bodies, emits `harnessed-<stack>` Dockerfile, host builds it
5. Surgical profile mount: only `.mcp.json` + `settings.json` mounted; image-baked skills survive; per-harness history surfaced via YAML manifests
6. All architecture documentation updated to new model; "isolated"/"transparent" narrative fully removed

**Key Decisions:**

- Recipes are Dockerfiles, not typed-YAML skill trees — assembler does not reinterpret install/layout
- We version the recipe's tag, not the vendor's data — pinned tag/SHA is the reproducibility unit
- Supply chain: pin + scan-the-image (not scan-vendored-deps) — gate = assembler rejects unpinned + osv-scanner V2 image scan + nightly rescan
- Harness parameterized in recipe Dockerfiles (`--host ${HARNESS}`) — assembler injects `ARG HARNESS`, refuses unsupported compositions
- Surgical mounts encoded in per-harness YAML manifests — path change = one-line manifest edit, not code search-and-replace

**Issues Resolved:**

- Typed-YAML recipe model replaced with Dockerfile model (simpler: run the framework's installer)
- Image lineage ambiguity resolved: 3 explicit layers (base → agent → stack)
- Profile dir-mount overwrote image-baked skills — fixed via surgical file-only mounts
- History surfacing without touching auth files — per-harness YAML manifests encode path inventories

**Issues Deferred:**

- Phase 10: opencode/codex home-folder investigation (MNT2-07) — no immediate stack support needed
- Phase 10: Combined capability test two-oracle approach (TST2-01, TST2-02, TST2-03) — v1 test remains active
